Zoom vào PM Agent — Vấn đề nó giải

Hình dung như đi tham quan 1 tòa nhà đã xây sẵn

Bạn cần dựng lại tòa nhà giống hệt — nhưng kiến trúc sư cũ đã đi mất, bản vẽ cũng không còn. Bình thường: cử 1 nhóm tới đo từng phòng, vẽ tay, viết note, rồi hy vọng họ không sót gì.

re-spec-mobile tự động hoá việc đó cho 1 app Android. Nó đi từng "phòng" (screen), đo từng "đồ vật" (UI element), ghi đầy đủ vị trí + nhãn + tương tác, rồi 1 AI khác viết lại thành tài liệu rebuild được — vẫn có 1 PM (con người) duyệt 3 chặng để đảm bảo không lệch ý.

Trước đây

Reverse-engineer tay

1 nhóm test ngồi mở app, screenshot, gõ Word, hy vọng nhớ hết. 2-3 tuần / feature. Thiếu sót khắp nơi. Không reproduce được.

Bây giờ

Workflow nửa-tự-động

Agent AI đi cả flow trong 30 phút. Capture đủ JSON + ảnh + nav graph. AI khác viết prose. PM ký 3 gate. 1 ngày ra spec hoàn chỉnh.

Đầu ra dùng cho

Cross-platform rebuild

Spec viết platform-agnostic. Coding agent có thể dựng lại trên Android, iOS, TypeScript, Flutter — chỉ dịch cú pháp.

Output — 6 layer

Mỗi feature đẻ ra 5 file, mỗi app có thêm 1 file tổng

Mỗi layer trả lời 1 câu hỏi khác nhau. Layer thấp = sự thật trần trụi (toạ độ, bounds). Layer cao = trừu tượng cho stakeholder. Ngoài cùng = platform-agnostic cho cả iOS rebuild.

Scope <feature>_scope.md
Gate 1 · PM ký
Cluster nào IN scope, cluster nào OUT, mỗi cluster có must_visit + optional_visit. PM trả lời mọi unknown trước khi agent chạy.
Giống "đề bài" thầu xây dựng ký với chủ nhà trước khi đào móng: "phòng nào sẽ làm, phòng nào không".
1
Observations <feature>_observations.md
Raw — sự thật trần
Mỗi screen 1 bảng bounds 5 cột (Class | Bounds | Clickable | Text | Content-desc) + Hành vi quan sát + Transitions. Không paraphrase, mọi text UI giữ nguyên văn.
Như note đo đạc: "góc Đông-Bắc, cửa sổ 1m × 1.2m, kính trong, có chốt". Không giải thích vì sao, chỉ ghi gì có.
2
Flow <feature>_spec.md
Block A→L · Mermaid
Chia feature thành 5-15 block logic (cụm screen cùng mục đích). Mermaid nav graph + state machine + UX bug đã quan sát. Không code, chỉ flow.
Bản đồ "đi từ phòng A sang phòng B bằng cửa nào, qua hành lang gì, mở chìa nào".
3
Implementation <feature>_feature_spec.md
9 section · Kotlin illustrative
Contract rebuild: state class, API + JSON Schema, data model (Kotlin chỉ là cú pháp minh hoạ — coding agent tự dịch sang Swift/TS/Dart), invariant, open question, acceptance criteria có ID.
Bản vẽ kỹ thuật cho thợ: "tường gạch đặc 220mm, cốt thép Ø8 a200". Cú pháp Kotlin chỉ là "đơn vị mm" — thợ Mỹ tự đổi sang inch.
4
Coverage Report <feature>_coverage_report.md
Gate 2 · auto + PM ký
Tool tự diff scope contract vs nav graph thực: captured / gaps / drift. PM duyệt pass/fail. Fail → loop về Phase 2.
Báo cáo nghiệm thu: "đã đo 8/10 phòng theo hợp đồng, sót phòng X vì lý do Y". Chủ nhà ký OK hoặc bắt làm lại.
5
App Overview spec/app_overview.md
Platform-agnostic · idempotent
1 file / app, tổng hợp đa-feature. 10 section auto-generated (sitemap, API surface, design system reuse map...) + 5 section prose designer/PM viết (mục tiêu, UX pattern, content rules...). Linter cấm token framework-specific.
Bản giới thiệu cho khách thuê: "tòa nhà 12 tầng, sảnh chính ở tầng 1, thang máy ở góc Đông". Không nói gạch, xi măng, sắt thép.
Workflow — 11 phase · 3 gate

Loop chạy bán-tự-động, PM khoá 3 chặng để không lệch ý

AI capture cái nó nghĩ ≠ cái PM muốn — đó là failure mode #1 của auto-test. Gate là 3 chặng PM bắt buộc ký artifact markdown trước khi sub-agent kế tiếp được phép chạy.

Phase 0 Bootstrap (1 lần / project)
Lệnh re-spec-init tự dò viewport + app đang focus → sinh .spec-profile.yml + scaffold folder spec/.
Như set up công cụ đo trước khi tới công trường.
Phase 1 Kickoff
Đọc profile, verify device + Portal + app focus, hỏi user feature nào, tạo todo list.
Gate 1 · Phase 1.5 Scope contract — PM sign-off
Claude propose draft scope (cluster IN/OUT, must_visit, unknowns). PM trả lời inline hoặc qua Telegram. Status flip draft → signed_off. Capture agent từ chối start nếu chưa ký.
PM nói "tôi muốn capture đúng 5 phòng này, không đụng 2 phòng kia" — ký vào hợp đồng trước.
Phase 2 Capture loop → agent app-explorer
Sub-agent (Sonnet) drive device: tap toạ độ centre-bounds, swipe, BACK, capture screenshot + a11y dump + cập nhật nav_graph.json.
"Thợ đo" đi từng phòng, chụp ảnh, ghi kích thước, đánh dấu lên sơ đồ.
Phase 3 Stuck handoff (khi BACK bị nuốt, modal trap...)
Không có Telegram → console hỏi user. Có Telegram → re-spec-pm-ask-stuck post 4 option (action / manual / skip / abort). Auto-skip nếu cùng screen đã skip ≥3 lần.
Phase 4 Coverage check (giữa loop, tự động)
re-spec-coverage-check --scope tìm clickable chưa tap, phân loại theo bucket scope. Loop tiếp Phase 2 nếu còn must_visit thiếu.
Gate 2 · Phase 4.5 Coverage report — PM audit
Tool auto-gen file diff scope vs reality. PM reply pass/fail qua Telegram → status tự flip. Fail → loop Phase 2.
Nghiệm thu xây thô: PM check "đã đo đủ không", duyệt tiếp hoặc bắt làm lại.
Phase 5 Spec writing → agent spec-writer
Sub-agent (Opus) đọc dump + nav graph + canonical sample → viết 3 file markdown (observations + flow + implementation). Trả DONE-PENDING-REVIEW kèm Open Question.
"Kiến trúc sư" ngồi viết bản vẽ kỹ thuật từ ghi chép của thợ đo.
Gate 3 · Phase 5.5 Spec review — PM trả lời Open Question §7
PM annotate **PM answer**: ... inline. Writer chạy lại mode=revise fold câu trả lời vào §3/§4/§5/§6/§8, xoá khỏi §7. Loop đến khi remaining_open_questions == 0.
Kiến trúc sư hỏi: "phòng tắm có vách kính hay rèm?". PM trả lời từng câu, kiến trúc sư cập nhật bản vẽ.
Phase 6 Build graph + validate
re-spec-build-graph parse mọi md → nodes.json + edges.json + index.json. re-spec-validate bắt V1-V9 errors.
Phase 7 Commit — luôn manual
Skill KHÔNG auto-commit. Show draft message, user duyệt rồi git commit.
Phase 8 App overview synthesis (≥2 feature, idempotent)
re-spec-app-overview render Layer 5. 10 auto section bọc HTML marker; prose ngoài marker preserve qua rerun. Linter cấm token Compose/Kotlin/SwiftUI...
Sau khi đã spec ≥2 feature, sinh tài liệu giới thiệu tổng app cho stakeholder. Mỗi lần thêm feature mới chỉ refresh phần tự động, phần prose designer viết vẫn còn nguyên.
Phân vai — 2 sub-agent

Một "thợ đo", một "kiến trúc sư" — tách bạch trách nhiệm

Orchestrator skill chỉ xử Phase 0/1/3/4/6/7. Phase 2 (capture) và Phase 5 (viết prose) delegate hẳn cho sub-agent để giữ main context gọn — và để mỗi việc đúng model phù hợp.

Capture pipeline

1 màn hình → 5 artifact trong ~1 giây

Mỗi lần re-spec-capture chạy, nó đi qua 5 bước tuần tự. Có fallback uiautomator dump tự kích hoạt khi Portal a11y tree rỗng (commit e01f02f) — nhờ thế WebView + React Native hybrid app vẫn capture được.

nav_graph

Append-only JSON: screens{} + edges[] mỗi feature

Vì sao bỏ qua droidrun.AdbTools.get_state()

Portal v0.6.x trả {"className":..., "bounds":"x,y,x,y"} còn droidrun 0.3.x parse schema cũ {"class_name":..., "bounds":{...}} → version mismatch. Toolkit này gọi raw content query rồi normalize trong process — bypass mismatch hoàn toàn.

Spec graph · MCP

Mọi spec file là 1 node + 1 set edge — query được như database

YAML frontmatter trong mỗi .md khai báo anchor + relationship. Tool build_graph.py parse hết → emit nodes.json + edges.json + index.json. MCP server stdlib-only expose 8 tool qua JSON-RPC 2.0 — Claude Code trong project tự load.

Anchor format

Bất di bất dịch

<feature>/<type>/<name>, lowercase snake_case, unique toàn repo. Validator V7 từ chối nếu sai format.

11 node type

Đa dạng entity

feature, layer, screen, block, component, api, data_model, criterion, invariant, question, state, cluster

14 edge type

Quan hệ chặt

navigates_to, belongs_to_block, renders_component, reuses_component, returns_model, gap, drift, verifies_scope, …

PM bridge — Telegram

PM duyệt gate qua điện thoại, không cần mở máy tính

Stdlib urllib only, không thêm dep. Bot post Open Question lên Telegram, PM reply trực tiếp message bot → re-spec-pm-sync long-poll getUpdates, fold reply vào đúng file/đúng anchor. Token chỉ qua env var, không vào yml.

Layer 5 — App Overview

1 tài liệu duy nhất cho cả app, platform-agnostic

Sau khi ≥2 feature đã commit, lệnh re-spec-app-overview tổng hợp ra spec/app_overview.md — dùng được cho cả iOS rebuild lẫn stakeholder không kỹ thuật. Mỗi lần rerun, phần auto refresh từ graph mới; phần prose designer/PM viết được preserve nguyên.

10 section auto-generated — bọc <!-- AUTO:KEY START/END -->

  • IDENTITY — bảng app name, package, viewport, locale, stack auto
  • INVENTORY — bảng feature × số screen / block / API / data model auto
  • SITEMAP — mỗi feature 1 bảng anchor → label → hash → capture file auto
  • CROSS_NAV — Mermaid flowchart edge cross-feature auto
  • REUSE_MAP — ứng viên design system (reuse_key xài chéo ≥2 feature) auto
  • API_SURFACE — bảng method × path × feature × returns auto
  • DATA_MODELS — bảng anchor × name × feature auto
  • INVARIANTS — bảng cross-feature invariant auto
  • OPEN_QUESTIONS — roll-up Open Q từ mọi feature auto
  • ACCEPTANCE — roll-up AC từ mọi feature auto

5 section prose — designer / PM tự viết

  • A. Mục tiêu sản phẩm prose
  • B. Navigation model (tab, modal, back, deep-link abstract) prose
  • C. UX state pattern (empty / loading / error / paywall / gating) prose
  • D. Content / copy rules (tone, locale, format, RTL) prose
  • E. Cross-cutting design decisions prose

Mạnh ở đâu, yếu chỗ nào

Mạnh

  • Filesystem-based + idempotent — crash giữa phase recover được; mỗi phase rerun an toàn.
  • Schema-validated 9 check — graph + MCP tự rebuild khi md mtime mới hơn nodes.json.
  • Cross-platform output — Layer 3 Kotlin illustrative + mapping table → iOS/TS/Dart rebuild không cần regen spec.
  • Telegram async PM gate — khoá scope drift nhưng không block dev work.
  • WebView/React Native hỗ trợ — fallback uiautomator dump tự kích hoạt (verified live trên Hailuo ComposeContainerActivity + RNContainerActivity).
  • 1 dep duy nhất — pyyaml. Telegram + MCP đều stdlib. Cài pip install 1 giây.
